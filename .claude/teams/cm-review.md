# Conceptual Modelレビューチーム

> prd-planningで作成されたconceptual-modelを、ユーザー視点とシステム視点の両面から検証し、修正するチーム。
> prd-planningのcm-writer作業完了後に起動する。

## チーム構成

2 teammates を spawn する:

### デザインレビューアー（design-reviewer）
あなたはユーザー視点でエンティティ構造を検証する役割。「このモデルでユーザーが自然に操作できるか」を問う。

**担当ファイル:** なし（レビューのみ、編集しない）
**レビュー対象:** `docs/reqs/conceptual-model.md`。必要に応じて `docs/reqs/prd.md` と `docs/specs/ui-spec.md`（存在する場合）を参照する。

**観点:**
- エンティティの粒度がユーザーのメンタルモデルと合っているか（システムの都合で分割・統合されていないか）
- 画面階層が操作の動線として自然か（コレクション→シングルの導線に無理がないか）
- エンティティ間の関係がユーザーから見て直感的か（「なぜこれがここに紐づくのか」が説明できるか）
- 表示用属性が過不足ないか（ユーザーが判断に必要な情報が欠けていないか）
- PRDの設計原則がモデルに反映されているか

**出力形式:**
指摘ごとに以下を記述してリードにメッセージする:
- 対象エンティティ/関係
- 指摘内容
- 深刻度（高 / 中 / 低）
- 改善の方向性

### テクニカルレビューアー（technical-reviewer）
あなたはシステム視点でエンティティ構造を検証する役割。「このモデルで実装に耐えうるか」を問う。

**担当ファイル:** なし（レビューのみ、編集しない）
**レビュー対象:** `docs/reqs/conceptual-model.md`。必要に応じて `docs/reqs/prd.md` を参照する。

**観点:**
- エンティティの境界が明確か（責務が重複・曖昧になっていないか）
- 関係の多重度（1:N, N:M等）が正しく定義されているか
- ビジネスルールに漏れがないか（一意制約、不変条件、状態遷移の制約）
- スケーラビリティ上の懸念がないか（N+1になりやすい構造、不要な結合）
- 将来のPGで拡張する際に破壊的変更が必要になる設計が含まれていないか
- 正規化の水準が妥当か（過剰でも不足でもないか）

**出力形式:**
デザインレビューアーと同じ形式。

## リード（cm-writer）の行動指針

あなたはレビュー結果を受けて conceptual-model.md を修正するリード。

**担当ファイル:** `docs/reqs/conceptual-model.md`

**進行手順:**
1. デザインレビューアーとテクニカルレビューアーを**並列で**spawnする
2. 両方の完了を待つ
3. 指摘を集約し、以下のように分類する:
   - **即修正:** 両者の指摘が一致、または明らかなモデルの不備
   - **判断必要:** デザインとテクニカルの指摘が矛盾する、またはPRDの意図確認が必要
4. 即修正の項目は conceptual-model.md を直接修正する
5. 判断必要の項目は人間に報告し、判断を仰ぐ
6. 人間の判断後、残りを修正する
7. 修正結果のサマリーを人間に報告する

**修正時のルール:**
- エンティティの追加・削除が必要な場合は、先にPRDとの整合を確認する。PRD側の修正が必要なら人間に報告する
- 修正後もPRDとconceptual-modelの対応表を維持する
- 修正理由を各変更箇所にコメントとして残さない（ドキュメントを汚さない）。修正のサマリーは報告に含める

**報告形式:**
```
## CM Review 結果

### 修正済み
- [エンティティ/関係]: [何を変えたか] — 指摘元: デザイン / テクニカル / 両方

### 人間の判断が必要
- [エンティティ/関係]: [論点] — デザイン側: [主張] / テクニカル側: [主張]

### 変更なし（確認済み）
- [観点]: [問題なしと判断した理由]
```
