# 実装チーム（impl）

> `docs/specs/impl-plan.md`（`.claude/rules/impl-planning.md` ワークフローで生成）のタスクカードに基づき、ストーリー単位で実装・テストを行うチーム。
> 2つのモードを行き来しながら進める。
> **ホワイトボードパターン**で知見を共有し、分業ではなく協働を実現する。

## チーム構成

| 役割 | モデル | 担当 |
|---|---|---|
| **lead: UIデザイナー** | Opus | モード間の統括、ホワイトボード管理、Spec差分判断、UI/UXレビュー |
| **teammate: フロントエンド** | Sonnet | 画面実装 |
| **teammate: バックエンド** | Sonnet | DB構築・API実装・インフラ設定 |
| **teammate: QA** | Sonnet | テスト作成・実行 |

## ファイル所有

| 役割 | 所有ファイル |
|---|---|
| lead | `docs/specs/*`（参照・更新）, `docs/reqs/*`（参照。企画変更が必要な場合は中断してユーザーに報告）, **ホワイトボードファイル**（作成・更新） |
| フロントエンド | `frontend/`（テストファイルを除く） |
| バックエンド | `backend/`（`backend/tests/` を除く）, `docker-compose.yml`, `Makefile` |
| QA | `backend/tests/`, `frontend/**/*.test.*`, `frontend/**/*.spec.*` |

## ホワイトボード

ストーリー実装の開始時にLeadが作成する共有Markdownファイル。各エージェントが発見・判断・申し送りを書き込み、モードをまたいで知見を引き継ぐ。

### なぜ必要か

Agent Teamsでは各エージェントが独立したコンテキストを持つ。SendMessageだけでは横断的洞察が失われ、モード切替時にコンテキストが途切れる。ホワイトボード（共有ファイル）を置くことで:

- FE/BEの並列実装中にAPI境界の発見を相互参照できる
- 実装Mode → テストMode切替時にQAが実装の文脈を把握できる
- 横断的な観察（性能・セキュリティ・設計判断）が記録として残る

### ファイルパスと命名

```
docs/whiteboard/{US-ID}.md
```

例: `docs/whiteboard/US-01.md`

### テンプレート

Leadがストーリー実装開始時に以下を作成する。

```markdown
# Whiteboard: {US-ID} {ストーリー名}

## Goal
{このストーリーで達成すること — user-stories.md の受け入れ条件を要約}

## Team Structure
- Lead（UIデザイナー）: Spec差分判断、UI/UXレビュー、ホワイトボード管理
- FE: {このストーリーでのFE担当タスク}
- BE: {このストーリーでのBE担当タスク}
- QA: Specs整合性テスト + シナリオテスト

## How Our Work Connects
- **API境界**: {FE/BEが共有するエンドポイントと契約}
- **データフロー**: {画面 → API → DB の流れで注意すべき点}
- **認証・認可**: {このストーリーに関わるアクセス制御}

## Key Questions
- {FEとBE両方の視点が必要な問い 1}
- {FEとBE両方の視点が必要な問い 2}
- {実装とテスト両方の視点が必要な問い 3}

---

## FE Findings
{フロントエンドの発見・判断・実装メモ}

### BEへの申し送り
{BEの実装に影響しうる発見}

---

## BE Findings
{バックエンドの発見・判断・実装メモ}

### FEへの申し送り
{FEの実装に影響しうる発見}

---

## Cross-Cutting Observations
{FE/BEの横断的な観察。どちらが書いてもよい}

---

## Test Findings
{テストMode移行後にQAが記入}

### Specs整合性テスト結果
{Pass/Fail と詳細}

### シナリオテスト結果
{Pass/Fail と詳細}

---

## Lead Decisions
{Leadが下した判断の記録 — Spec修正、UI/UXフィードバック、モード切替判断など}
```

### 運用ルール

1. **Leadが作成・管理する** — ホワイトボードの作成と、teammateの発見の転記はLeadの責務。teammateがファイル書き込みできない場合、SendMessageで受け取った内容をLeadが転記する
2. **各エージェントは自分のセクションに書く** — FEは「FE Findings」、BEは「BE Findings」、QAは「Test Findings」に記入する
3. **申し送りセクションを活用する** — 相手の担当領域に影響する発見は「〜への申し送り」に記入する。自分の担当範囲を超えた予測・提案を積極的に書く
4. **Cross-Cutting Observationsは誰でも書ける** — 性能・セキュリティ・設計一貫性など、単一の担当領域に収まらない観察を記録する
5. **モード切替時にLeadが引き継ぎを書く** — 実装Mode → テストMode 移行時、Leadは「Lead Decisions」に実装Modeの要約を記入し、QAがコンテキストを把握できるようにする

### ライフサイクル

- **git管理**: `docs/whiteboard/` はコミット対象。ストーリーの実装PRにホワイトボードファイルも含める
- **完了後の扱い**: ストーリー完了後も参照資料として保持する。ホワイトボードの行数はPR diff上限（+100行）の計算対象外とする
- **クリーンアップ**: PG完了時に、該当PGのホワイトボードファイルをまとめて削除してよい

## モード構成

同時稼働は常に3人以内。4人を2モードに分割して運用する。

### 実装Mode（Lead + FE + BE）

**目的:** ストーリーの実装を完了させる

**Leadの責務:**
- **ストーリー開始時にホワイトボードを作成する**（`docs/whiteboard/{US-ID}.md`）
- `docs/specs/impl-plan.md` のタスクカードからFE/BEへの作業指示を出す
- api-spec.md を参照し、FE/BE間のAPI契約をホワイトボードに記入して共有する
- 実装中にSpecとの齟齬を検知した場合:
  - 企画ドキュメント（reqs/）の変更が不要な軽微なSpec齟齬 → Leadの判断でSpecsを先に修正してからFE/BEに実装指示を出す。判断理由をホワイトボードの「Lead Decisions」に記録する
  - 企画ドキュメント（reqs/）の変更が必要な場合 → 実装を中断してユーザーに報告する
- FE/BEからのSendMessageで受け取った発見・申し送りをホワイトボードに転記する
- FE/BEの進行を管理し、必要に応じて追加指示を送る

**フロントエンドの責務:**
- ui-spec.md + product-model.json（screens）に基づいて画面を実装する
- api-spec.md のエンドポイント定義に合わせてAPIクライアントを実装する
- analytics-spec.md に基づいてフロントエンドのイベント計測を実装する
- **実装中の発見・判断・BEへの申し送りをLeadに報告する**（Leadがホワイトボードに転記）

**バックエンドの責務:**
- db-schema.md に基づいてマイグレーション・モデルを実装する
- api-spec.md に基づいてエンドポイントを実装する
- auth-spec.md に基づいて認証・認可を実装する
- infra-spec.md を参照し、必要に応じて docker-compose.yml / Makefile を更新する
- analytics-spec.md に基づいてバックエンドのイベント計測を実装する
- **実装中の発見・判断・FEへの申し送りをLeadに報告する**（Leadがホワイトボードに転記）

### テストMode（Lead + QA）

**目的:** 実装の品質を機能面・体験面の両方から検証する

**Leadの責務（モード切替時）:**
- ホワイトボードの「Lead Decisions」に実装Modeの要約を記入する
- FE/BEの申し送りと横断的観察をQAに共有する（ホワイトボードを読むよう指示）

**QAの責務:**
- **ホワイトボードを読んでから**テストを開始する（実装の文脈を把握）
- `.claude/rules/test-execution.md` のワークフローに従ってテストを実行する
  - セクションA（Specs整合性テスト）: DB/API/AuthにFailがある場合は中断してLeadに報告
  - セクションB（シナリオテスト）: 優先度順（Must → Should → Could）に実行
- テスト実行中にSpecや実装の修正はしない。Failとして記録し報告する
- **テスト結果をホワイトボードの「Test Findings」に記入する**（またはLeadに報告して転記）
- ホワイトボードのCross-Cutting Observationsに書かれた懸念事項も検証対象とする

**Leadの責務（判定時）:**
- UI/UX観点でレビューする（デザインシステムとの整合性、操作の流れ、フィードバックの分かりやすさ）
- QAのテスト結果とUI/UXレビュー結果を統合する
- 判定:
  - Fail or UI/UXフィードバックあり → 実装Modeに戻し、FE/BEに修正指示。ホワイトボードに修正理由を記録
  - All OK → 完了、Spec差分レポートを出力

## フロー

```
┌─────────────────────────────────────┐
│ Lead: ホワイトボード作成              │
│  docs/whiteboard/{US-ID}.md          │
│  - Goal, Team, 接続点, Key Questions │
└──────────┬──────────────────────────┘
           ↓
┌─────────────────────────────────────┐
│ 実装Mode（Lead + FE + BE）           │
│  - FE/BE 並列実装                    │
│  - FE/BE → Lead → ホワイトボード転記  │
│  - Lead: Spec差分の判断・修正、指示    │
│  ※企画変更が必要な場合は中断→ユーザーへ │
└──────────┬──────────────────────────┘
           ↓ 実装完了
┌─────────────────────────────────────┐
│ Lead: モード切替の引き継ぎ            │
│  - 実装Modeの要約をホワイトボードに記入 │
│  - QAにホワイトボード参照を指示        │
└──────────┬──────────────────────────┘
           ↓
┌──────────┴──────────────────────────┐
│ テストMode（Lead + QA）              │
│  - QA: ホワイトボード参照 → テスト実行 │
│  - QA: Test Findingsに結果記入       │
│  - Lead: UI/UXレビュー               │
└──────────┬──────────────────────────┘
           ↓ Fail or UI/UXフィードバックあり
           → ホワイトボードに修正理由記録 → 実装Modeに戻る
           ↓ All OK
┌──────────┴──────────────────────────┐
│ Lead: Spec差分レポートを出力          │
│  - 実装中に変更したSpecsの一覧        │
│  - 変更理由と影響範囲                 │
└──────────┬──────────────────────────┘
           ↓
  次のストーリーの指示を待つ
```

## Spec差分レポートの形式

```
## Spec差分レポート
### 変更したSpecs
| ファイル | 変更前 | 変更後 | 理由 |
|---|---|---|---|
| {ファイル} | {変更前} | {変更後} | {理由} |

### 影響範囲
- {影響の説明}

### 未反映の課題
- {あれば記載}
```
