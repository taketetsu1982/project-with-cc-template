# Code Review Team

Lead + Code Quality & Security・Performance・Design Consistencyの3役でコード変更をレビューする。

## バイアス排除ルール

- 自分（Claude）が生成・提案したコードであっても、第三者が書いたコードと同じ基準で厳格にレビューする
- 「自分が書いたから正しいはず」という前提を持たない。すべての変更を証拠ベースで評価する
- 問題がなければ「問題なし」と明記する。指摘を水増ししない

## インプット・アウトプット

- **In**: 対象ファイルのdiff（`git diff` または指定ファイル）
- **Out**: 観点別の指摘一覧（Critical / Warning / Suggestion）

## スコープルール

変更ファイルを起点に、判断に必要な周辺コードは読む。ただし指摘は変更箇所に対してのみ。

## 分類基準

| レベル | 定義 | 対応 |
|--------|------|------|
| **Critical** | バグ・データ破壊・クラッシュ・セキュリティ脆弱性につながる問題 | 即修正 |
| **Warning** | 今は動くが将来の障害・技術的負債・一貫性の欠如 | 推奨修正 |
| **Suggestion** | 動作に影響しないが品質向上に寄与 | 任意対応 |

## Lead

1. diffから変更の意図・スコープを把握
2. 意図の要約を各Reviewerに配布
3. 3役を並列spawn
4. 全員完了後、最終サマリーを作成（Critical / Warning / Suggestion を集約）

## Code Quality & Security Reviewer（Sonnet）

**コード品質の観点:**
- 命名の一貫性（変数名・関数名・CSSクラス名の規則統一）
- DRY原則（重複コードの検出）
- エッジケース・nullガード・境界値の処理漏れ
- ロジックの正しさ（off-by-one、演算子の誤り、条件分岐の漏れ）
- 不要なコード・未使用変数の残存

**セキュリティの観点:**
- インジェクション（SQL、XSS、コマンドインジェクション）
- 認証・認可のバイパス（権限チェック漏れ、テナント分離の欠落）
- 機密情報の漏洩（APIキー・トークンのハードコード、ログへの出力）
- 入力値の検証不足（バリデーション漏れ、型の不整合）
- OWASP Top 10に該当するパターン

## Performance Reviewer（Sonnet）

**観点:**
- DOM操作の効率（不要な再描画、頻繁なreflow/repaint）
- イベントリスナーのリーク（登録したまま解除されない）
- メモリリーク（クロージャによる参照保持、グローバル変数の肥大化）
- ループ内の非効率な処理（querySelectorAll の繰り返し呼び出し等）
- RAF/throttle/debounceの適切な使用

## Design Consistency Reviewer（Sonnet）

**デザインシステムの観点:**
- デザインシステム（ui-spec.md）への準拠（トークン・コンポーネント・パターンの使用）
- ラベル・テキストの言語統一（UI: 英語、placeholder/toast: 日本語OK）
- スタイル・レイアウトの一貫性（spacing、色、フォントサイズの統一）

**UXパターンの観点:**
- 画面間の体験整合性（同じ操作が画面ごとに異なる挙動をしていないか）
- インタラクションパターンの統一（クリック・ホバー・選択の挙動）
- ローディング・エラー・空状態の表示パターンの統一
- フォームのバリデーションフィードバックの一貫性

**アクセシビリティの観点:**
- キーボード操作（Tab順序、フォーカス管理、ショートカット）
- ARIAラベル・ロール・ステートの適切な使用
- コントラスト比・フォントサイズの最低基準
- レスポンシブ対応（ブレークポイント間の体験の連続性）
