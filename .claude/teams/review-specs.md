# Specsレビューチーム（Team B）

> 実装前にSpecs群の設計問題を潰すチーム。
> アウトプットは「実装前に解決すべき問いのリスト」（優先順位・判断オーナー付き）。

## バイアス排除ルール

- 自分（Claude）が生成・提案したSpecsであっても、第三者が作成したSpecsと同じ基準で厳格にレビューする
- 「自分が書いたから正しいはず」という前提を持たない。すべての設計判断を証拠ベースで評価する
- 問題がなければ「問題なし」と明記する。指摘を水増ししない

## Review Gate

リードがteammateをspawnする前に確認する。1つでもNoならSpecs補完が先。

- [ ] ユーザー課題（Why）と機能の振る舞い（入力→処理→出力）が書かれているか
- [ ] 対象のスキーマ／APIと実装方針があるか
- [ ] スコープ外が明示されているか

## 重要度の定義

| レベル | 定義 | 対応 |
|---|---|---|
| **Critical** | このまま実装すると手戻りが確定する。実装着手前に解決必須 | 即修正 |
| **Warning** | 実装は可能だが、放置すると問題化する | 推奨修正 |
| **Suggestion** | 現スコープでは許容。次回以降の検討事項 | 任意対応 |

## チーム構成

### 技術設計レビューアー（technical-reviewer）
**レビュー対象:** `docs/specs/db-schema.md`, `docs/specs/api-spec.md`, `docs/specs/auth-spec.md`, `docs/specs/infra-spec.md`

**データ設計の観点:**
- テーブル定義がconceptual-modelのエンティティと一致しているか
- 正規化の水準が妥当か（過剰でも不足でもないか）
- インデックス・制約の過不足

**API設計の観点:**
- エンドポイントがリソース指向で一貫しているか
- レスポンス構造に不要なデータが含まれていないか（過剰露出）
- ページネーション・フィルタリングの設計が妥当か

**セキュリティ設計の観点:**
- 認可モデルに穴がないか（権限チェック漏れ、権限昇格の経路）
- テナント分離の設計が全エンドポイントで一貫しているか
- スコープ外のリソースへのアクセスが404で返る設計になっているか（存在の開示防止）
- 機密データの保存・送信方針が定義されているか（暗号化、マスキング）

**インフラ設計の観点:**
- 検証環境の構成がspecs群の要件を満たしているか
- DB・APIサーバー・フロントエンドの接続構成に矛盾がないか
- 認証・認可の構成がauth-specと整合しているか
- 環境変数・シークレット管理の方針が定義されているか
- CI/CDパイプラインがtest-specのテスト実行要件を満たしているか

### 機能設計レビューアー（functional-reviewer）
**レビュー対象:** `docs/specs/` 全体、`docs/reqs/prd.md` との整合

**観点:** 仕様の完全性、ユーザー体験、スコープの過不足

### 統合レビューアー（integration-reviewer）
**起動条件:** 技術設計・機能設計の**両方完了後**にspawnする。

**観点:**
- 技術設計と機能仕様の整合
- 両レビューで同一箇所に指摘 → リスク引き上げ
- 実装着手のための優先順位決定

## リードの行動指針

1. Review Gateを確認
2. 技術・機能レビューアーを**並列**spawn
3. **両方完了後**に統合レビューアーをspawn
4. 統合結果を人間に報告
5. Criticalがゼロになったら実装着手を承認
