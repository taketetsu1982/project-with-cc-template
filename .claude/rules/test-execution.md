- テストファイルの配置規約: {プロジェクトに合わせて記述}
- テスト実行コマンド: {プロジェクトに合わせて記述}

## 実行ワークフロー

> `docs/specs/test-spec.md` に基づいて実装の検証を行う。
> 「テストして」「テスト実行」等の指示で起動する。

## 前提確認

1つでもNoなら人間に報告して中断。

- [ ] `docs/specs/test-spec.md` が存在し、内容が記述されているか
- [ ] テスト対象の実装が完了しているか

## Step 1: Specs整合性テスト（test-spec セクションA）

test-spec.mdのセクションAに定義された項目を上から順に実行する。

1-1. **DB Schema整合性** — テーブル定義と照合
1-2. **API整合性** — 全エンドポイントに正常系/異常系リクエスト
1-3. **認証・認可整合性** — 認証フロー、ロール別許可/拒否、テナント分離
1-4. **UI整合性** — 全画面パス、状態表示、フォームバリデーション
1-5. **計測整合性** — イベント発火、プロパティ照合

**Step 1完了時の出力:** カテゴリ別のPass/Fail集計 + Fail詳細

**Step 2への遷移判断:**
- Failが0件 → Step 2に進む
- DB/API/AuthにFail → 中断して報告（シナリオテストの前提が崩れる）
- UI/Analyticsのみ → Step 2に進む（Failは報告に含める）

## Step 2: シナリオテスト（test-spec セクションB）

test-spec.mdのセクションBを優先度順に実行する。

実行順序:
1. Must（正常系 → エッジケース）
2. Should（正常系のみ）
3. Could（スキップ可）

## 最終レポート

```
## テスト実行レポート
### サマリー
- Specs整合性: N Pass / N Fail
- シナリオテスト: N Pass / N Fail
### Fail分類
| # | テスト項目 | 分類 | 説明 |
|---|---|---|---|
| 1 | [項目] | 実装バグ / Spec不備 / 新規エッジケース | [詳細] |
### 次のアクション
- 実装バグ: [修正対象]
- Spec不備: [修正が必要なSpec]
- 新規エッジケース: [追加候補]
```

## 注意

- テスト実行中にSpecや実装の修正はしない。Failとして記録し報告する
- 「次のアクション」は提案のみ。実行は人間の指示を待つ
