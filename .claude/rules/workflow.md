# 開発ワークフロー（アジャイル・ドキュメント駆動）

## 基本モデル: 三角形ループ

```
        reqs（企画）
       ↗        ↘
  実装  ⇄  specs（設計）
```

- reqs・specs・実装はどの頂点からでも着手してよい
- ある頂点を更新したら、他の2頂点との整合性をCCが検知・提案する
- 人間は意思決定とレビューに集中する。記述はCCが行う

### 旧ルールとの違い
- ~~実装だけ先に進めるのは厳禁~~ → プロトタイプ・スパイクから始めてreqsに反映するのは正当なフロー
- ~~reqs → specs → 実装の一方向~~ → 三角形の任意の辺を行き来する
- ~~手動で同期マップを辿る~~ → CCが差分検知して同期提案する

## 原則

- **必要なドキュメントだけ作る** — フィットジャーニーのフェーズごとに必要なドキュメントは異なる。`product-goals.md` のドキュメントマップで管理する
- **CCが生成し、人間が微調整する** — specsはCCが `product-model.json` 等から自動生成する。人間の意思決定部分は `<!-- human-decision -->` マーカーで保護する
- **不整合はCCが検知する** — 上流を変えたら下流をCCが自動チェックし、差分を報告する。人間が同期マップを暗記する必要はない
- **迷ったら実装を止めて確認する** — これは変わらない
- **不明な点は推測で実装せずTODOコメントを残す** — これも変わらない

## フィットジャーニーとドキュメントマップ

各フェーズで必要なドキュメントだけを作成する。不要なものはテンプレートすら置かない。

| フェーズ | 検証すること | 必要なドキュメント |
|---|---|---|
| **CPF** (課題発見) | 課題は実在するか | `pg{N}.md`, `mrd.md`（軽量） |
| **PSF** (課題解決) | この解き方で解決するか | + `prd.md`, `product-model.json`（entities/actors） |
| **SPF** (ソリューション) | プロダクトとして使えるか | + screens/transitions, `user-stories.md`, `ui-spec.md` |
| **PMF** (市場適合) | 市場に受け入れられるか | + `db-schema.md`, `api-spec.md`, `auth-spec.md`, `test-spec.md` |
| **GTM/Growth** | スケールするか | + `analytics-spec.md`, `infra-spec.md` |

- `product-goals.md` のドキュメントマップが正。上記は目安
- フェーズが進んでドキュメントが必要になったら、CCがテンプレートを生成して提案する
- 前フェーズのドキュメントは引き続き更新対象。ただし更新頻度は下がる

## CCによるドキュメント自動生成

### 自動導出spec（CCが `product-model.json` 等から生成）

| Spec | 生成元 | 人間の役割 |
|---|---|---|
| `conceptual-model.md` | `product-model.json` | 設計意図の補足（JSON側が正） |
| `db-schema.md` | `product-model.json`（entities） | 制約・インデックスの判断 |
| `api-spec.md` | entities + actors + screens | エンドポイント粒度の判断 |
| `auth-spec.md` | actors の touches | セキュリティ要件の判断 |
| `test-spec.md` | user-stories + specs | 優先度・網羅度の判断 |

### 意思決定spec（人間の判断が主、CCは骨格生成）

| Spec | 人間が決めること |
|---|---|
| `ui-spec.md` | デザイントークン、インタラクション、状態遷移 |
| `analytics-spec.md` | 何を計測するか、成功指標 |
| `infra-spec.md` | 構成選定、コスト判断 |

### 人間の意思決定の保護

CCがspecを再生成する際、`<!-- human-decision -->` 〜 `<!-- /human-decision -->` で囲まれた部分は上書きしない。

```markdown
## 認証方式
<!-- human-decision -->
JWTではなくセッションベース認証を採用する。理由: ...
<!-- /human-decision -->
```

## 作業の進め方

### どの頂点から始めてもよい

**パターンA: 企画から始める（通常）**
1. `pg{N}.md` でスコープ・出口条件を定義
2. ドキュメントマップに従い必要なreqsを作成
3. CCがspecsを自動生成 → 人間がレビュー
4. 実装

**パターンB: 実装から始める（プロトタイプ・スパイク）**
1. プロトタイプを実装して学びを得る
2. 学びをreqsに反映（CCが差分を提案）
3. CCがspecsを再生成

**パターンC: specsから始める（技術的制約が先に判明）**
1. 技術調査の結果をspecsに記述
2. specsの制約をreqsに反映（CCが差分を提案）
3. 実装

### 三角形ループの回し方

1. **変更する** — 任意の頂点のドキュメントまたはコードを変更する
2. **CCが検知** — 変更された頂点から、他の2頂点への影響をCCが分析する
3. **差分を報告** — CCが「ここが古くなっている」「ここに矛盾がある」を報告する
4. **人間が判断** — 同期する・しない・後回しにするを決める
5. **CCが同期** — 人間の判断に基づいてCCがドキュメントを更新する

## レビューゲート

- レビューゲートの詳細は `review-gate.md` を参照
- **Specsレビュー**（必須）: 実装着手前にspecsの整合性をチェック
- **コードレビュー**（必須）: ストーリー実装完了後にコード品質をチェック
- フィールド照合チェックはCCが自動実行する（`doc-sync.md` 参照）

## 実装時の制約

- DBを変更する前に `product-model.json` のentitiesを確認する
- `product-model.json` にないエンティティをDBに追加しない。先にreqsに戻る
- APIを追加するときは `api-spec.md` の命名規則に従い、対応するPRD機能を明記する
- 画面を追加するときは `product-model.json` の screens を先に追加/更新する
- イベント計測を追加するときは `analytics-spec.md` のイベント命名規則に従う

## PG遷移（次のProduct Goalへ進む）

- **トリガー**: 現在のPGの出口条件をすべて満たしたとき
- **アクション**:
  1. 現在の `pg{N}.md` のステータスを `Done` に更新する
  2. `product-goals.md` のPG一覧で該当PGのステータスを `Done` に更新する
  3. `docs/reqs/pg{N+1}.md` を作成する（`pg{N}.md` の構造をコピーし、内容はテンプレートに戻す）
  4. `product-goals.md` のPG一覧に新しい行を追加する
  5. `product-goals.md` のドキュメントマップに新PGの行を追加する
  6. 新PGの企画からワークフローを再開する

## スキップのルール

- ドキュメントを書く前に `product-goals.md` のドキュメントマップを確認する
- 現在のフェーズで不要なドキュメントは作成しない
- 現在のPGに含まない機能は書かない・実装しない
- 既にドキュメントが最新なら更新不要。次のステップに進んでよい

## セッション終了時

- Done / Next / Risks-TODO を記録する
