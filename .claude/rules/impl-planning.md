# 実装計画ワークフロー

> Track A（Specs）と Track B（User Stories → Test Spec）の両方が完了した時点で自動起動する。
> 成果物は `docs/specs/impl-plan.md` に出力する。

## 前提確認

1つでもNoなら人間に報告して中断。

- [ ] `docs/reqs/user-stories.md` が記述されているか
- [ ] `docs/specs/` 配下のファイルが記述されているか（少なくともdb-schema, api-spec, ui-spec）

## Step 1: ストーリー単位のタスク分解

`docs/reqs/user-stories.md` を読み、各ストーリーに必要な実装を抽出する。

各ストーリーについて:
1. シナリオを読む
2. 必要なエンティティを `docs/reqs/conceptual-model.md` から特定する
3. 対応するSpecs（db-schema, api-spec, auth-spec, ui-spec, analytics-spec）の該当箇所を特定する
4. 実装対象ファイルとその内容を一覧にする

**タスクカードの形式:**
```
### US-[ID]: [ストーリー名]（[優先度]）
**ゴール:** このタスクが完了すると何が動くか（1文）
**実装対象:**
- DB / API / Auth / UI / Analytics / Test: [新規 or 変更]
**前のストーリーとの差分:**
- [既存への変更がある場合のみ記載]
```

## Step 2: 実装順序の提示

タスクカードを優先度順（Must → Should → Could）に並べて人間に提示する。

## 出力

タスクカードと実装順序を `docs/specs/impl-plan.md` に書き出す。
ファイルがすでに存在する場合は上書きする（specs と同様、reqs からの生成物として扱う）。

## Step 3: 人間の指示を待つ

人間がストーリーを選んだら、そのタスクカードの内容に従って実装を開始する。

## ストーリー実装時のドキュメント更新ルール

既存のテーブルやAPIを変更・作り直す必要が出た場合、**実装を変更する前に企画ドキュメントから更新する。**

更新順序（企画→設計）:
1. 変更の起点を特定する
2. 企画ドキュメントから更新する
3. 設計ドキュメントも更新する
4. ドキュメント更新が完了してから実装を変更する

**やってはいけないこと:**
- ドキュメントを更新せずに実装を変更する
- 設計だけ更新して企画を放置する
- 「あとでドキュメントを直す」と言って実装を先に進める

## ストーリー完了の確認

1. 対応するテストを実行する
2. Pass したら報告し、次のストーリーの指示を待つ
3. Fail したら修正してから報告する

## 注意

- ストーリー間で先読みしない
- テーブルやAPIの作り直しは正常。ただし必ずドキュメントを先に更新する
- 次のストーリー着手前にタスクカードの差分を再確認する
