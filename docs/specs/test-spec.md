## A. Specs整合性テスト

### DB Schema整合性

| ID | テスト項目 | 検証対象 | 検証方法 | 優先度 |
|---|---|---|---|---|
| DB-01 | テーブル存在 | users, oura_connections, daily_records, physical_scores, mental_entries | マイグレーション後に `\dt` で5テーブルの存在を確認 | Must |
| DB-02 | 共通カラム | 全テーブルの id, created_at, updated_at, deleted_at | 各テーブルで型・制約・デフォルト値を検証（id: UUID PK, created_at/updated_at: TIMESTAMPTZ NOT NULL DEFAULT NOW(), deleted_at: TIMESTAMPTZ NULLABLE） | Must |
| DB-03 | oura_connections カラム定義 | user_id, access_token, refresh_token, token_expires_at | user_id: UUID NOT NULL UNIQUE FK→users.id, access_token/refresh_token: TEXT NOT NULL, token_expires_at: TIMESTAMPTZ NOT NULL | Must |
| DB-04 | daily_records カラム定義 | user_id, record_date | user_id: UUID NOT NULL FK→users.id, record_date: DATE NOT NULL | Must |
| DB-05 | daily_records ユニーク制約 | (user_id, record_date) | 同一ユーザー・同一日付で2件目のINSERTが拒否されることを確認 | Must |
| DB-06 | physical_scores カラム定義 | daily_record_id, sleep_score, readiness_score, activity_score, hrv_rmssd, resting_heart_rate, sleep_stages | daily_record_id: UUID NOT NULL UNIQUE FK→daily_records.id, スコア系: INTEGER/FLOAT NULLABLE, sleep_stages: JSONB NULLABLE | Must |
| DB-07 | mental_entries カラム定義 | daily_record_id, mood, energy, stress, focus | daily_record_id: UUID NOT NULL UNIQUE FK→daily_records.id, 4軸: INTEGER NOT NULL CHECK(BETWEEN 1 AND 7) | Must |
| DB-08 | mental_entries CHECK制約 | mood, energy, stress, focus | 0以下・8以上の値がINSERTで拒否されることを確認 | Must |
| DB-09 | FK カスケード | 全外部キー | 親レコード削除時の振る舞いを確認 | Must |
| DB-10 | インデックス存在 | 全定義済みインデックス | oura_connections_user_id_idx, daily_records_user_id_idx, physical_scores_daily_record_id_idx, mental_entries_daily_record_id_idx の存在確認 | Must |

### API整合性

> api-spec.md v0.2 に基づく。

| ID | テスト項目 | 検証対象 | 検証方法 | 優先度 |
|---|---|---|---|---|
| API-01 | Oura OAuth2開始 | OAuth2認証フロー開始エンドポイント | リクエストでOura認可URLへのリダイレクトが返ることを確認 | Must |
| API-02 | Oura OAuth2コールバック | コールバックエンドポイント | 認可コード受取後にトークンが保存され、接続状態が更新されることを確認 | Must |
| API-03 | Oura接続状態取得 | 接続状態取得エンドポイント | 接続済み/未接続の状態が正しく返ることを確認 | Must |
| API-04 | Oura接続切断 | 切断エンドポイント | トークン削除後に204が返り、接続状態が未接続になることを確認 | Must |
| API-05 | メンタルエントリ作成 | POST mental_entries | mood/energy/stress/focus（各1-7）で201が返り、daily_recordに紐づくことを確認 | Must |
| API-06 | メンタルエントリ更新 | PATCH/PUT mental_entries | 既存エントリの値変更で200が返り、更新後の値が正しいことを確認 | Must |
| API-07 | メンタルエントリ バリデーション | POST mental_entries（不正値） | 範囲外の値（0, 8）、欠損フィールドで422が返ることを確認 | Must |
| API-08 | 日次サマリー取得 | GET daily_records/{date} | 指定日のPhysicalScore + MentalEntryが返ることを確認 | Must |
| API-09 | トレンドデータ取得 | GET /api/v1/trends | window=7/14/30のデータがリスト形式で返り、strain_recoveryフィールドが含まれることを確認 | Must |
| API-10 | レスポンス形式 | 全エンドポイント | Content-Type: application/json、日時: ISO 8601、ID: UUID v4 を確認 | Must |
| API-11 | エラーレスポンス | 全エンドポイント | `{ "detail": "..." }` 形式で返ることを確認 | Must |

### 認証・認可整合性

> PG1は単一ユーザー（SelfTracker）のみ。auth-spec.md v0.2 に基づく。JWT認証（Access Token 1時間 / Refresh Token 7日）。

| ID | テスト項目 | 検証対象 | 検証方法 | 優先度 |
|---|---|---|---|---|
| AUTH-01 | 認証必須エンドポイント | 保護対象の全エンドポイント | Authorization ヘッダーなしで401が返ることを確認 | Must |
| AUTH-02 | Ouraトークンリフレッシュ | OuraConnection のトークン有効期限切れ | 期限切れ時にリフレッシュトークンで自動更新されることを確認 | Must |
| AUTH-03 | スコープ制限 | PhysicalScore | PhysicalScoreへの書き込み操作（POST/PUT/DELETE）が拒否されることを確認（Rのみ許可） | Must |

## B. シナリオテスト

### S-01: Oura APIを接続する（US-01）

**テスト種別:** E2E
**優先度:** Must
**エピック:** Ouraデータ連携

**事前準備:** ユーザーが作成済み。Oura APIは未接続状態。Oura OAuth2モックサーバー準備。

**正常系**
| # | 操作 | 期待結果 | 検証方法 |
|---|---|---|---|
| 1 | Oura接続設定画面を開く | OAuth2未接続状態が表示される | UIに「未接続」表示を確認 |
| 2 | 「Ouraを接続」ボタンをタップ | Oura OAuth2認証画面にリダイレクト | リダイレクトURLを確認 |
| 3 | OAuth2認証画面で認可する | コールバックが処理される | コールバックエンドポイントへのリクエストを確認 |
| 4 | 認証完了 | 接続状態が「接続済み」に変わる | oura_connectionsテーブルにレコード存在・UIで接続状態表示 |
| 5 | 接続完了後 | フィジカルデータの日次取得が開始される | physical_scoresにデータが作成されることを確認 |

**エッジケース**
| # | 操作 | 期待結果 | 検証方法 |
|---|---|---|---|
| E1 | OAuth2認証をキャンセルする | 接続状態は変わらず、エラーメッセージ表示 | oura_connectionsにレコードなし・エラーメッセージ表示確認 |
| E2 | アクセストークン有効期限切れ後にAPI呼び出し | リフレッシュトークンで自動更新 | token_expires_atが更新・API呼び出し成功 |
| E3 | Oura APIが一時的に利用不可 | リトライまたはエラー状態を表示 | エラーメッセージ表示・接続状態の適切な更新 |
