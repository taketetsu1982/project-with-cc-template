## Issue 1: 「Aを変えたらBも直す」が書かれていない

**深刻度:** Critical（3回再発）
**影響:** `.claude/rules/workflow.md`

### どういう問題か

テンプレートは「reqs → specs → 実装」の流れを定めている。
しかし**上流を更新したとき、下流のどこを確認すべきか**が書かれていない。
結果として、変更が伝播せず、ドキュメント間の整合性が壊れる。

### 3回繰り返した

| 何が起きたか | いつ |
|------------|------|
| PRDで4軸を変更 → pg1.mdは旧軸のまま放置 | 3/19 |
| PRDに「睡眠ステージ比率」と明記 → DB/API/Testすべてに`sleep_stages`がなかった | 3/19 |
| auth-specに「メール+パスワード認証」→ DBの`users`テーブルに`email`も`password_hash`もなかった | 3/21 |

バージョン参照の陳腐化も同根。impl-planが「api-spec v0.2」を参照したまま、api-specはv0.4になっていた（3/21 KPT）。

### こうすれば防げた

**1. ドキュメント依存マップ**をワークフローに追加する。

| 更新したドキュメント | 確認が必要なドキュメント |
|---|---|
| prd.md | pg{N}.md, conceptual-model.md, user-stories.md |
| conceptual-model.md / product-model.json | db-schema.md, api-spec.md, ui-spec.md |
| api-spec.md | test-spec.md, impl-plan.md, ui-spec.md |
| auth-spec.md | db-schema.md, test-spec.md |
| user-stories.md | test-spec.md |

**2. フィールド照合チェック**をSpecs導出後に必須化する。

- PRD機能概要の全項目 → DB Schemaのカラムに存在するか
- DB Schemaの全カラム → API Specのレスポンスに存在するか
- API Specの全エンドポイント → Test Specに存在するか
- Auth Specの認証要件 → DB Schemaに反映されているか

## Issue 2: 抽象的な方針が実装レベルに落ちないまま進む

**深刻度:** Critical（3回再発）
**影響:** `.claude/rules/workflow.md`, `docs/reqs/prd.md`（テンプレート構造）

### どういう問題か

PRDの設計原則に「Strain/Recoveryは独立2軸で表示する」のような抽象方針を書く。
しかし「どのメトリクスで構成するか」「どう計算するか」「統計的に妥当か」を定義するステップがない。
抽象のまま下流に渡ると、Specs導出で**具体化できない・誤った具体化をする・ドキュメント間で解釈がズレる**。

### 3回繰り返した

| 何が起きたか | いつ |
|------------|------|
| 「Strain/Recovery独立2軸」→ 計算式なし → APIにフィールドが作られなかった | 3/19 |
| 計算式を追加 → StrainにActivity Scoreを含めたが「良い活動日=高負荷」と誤判定。APIは日次時系列でなく集約値で返し、グラフ描画不能。USの「空表示」とAPIの「404」が矛盾 | 3/21 |
| 「Pearson相関のpower 50-60%」で30日閾値設定 → 精密計算で44.5%と判明。自己相関考慮で30日でも不十分 | 3/21 |

### こうすれば防げた

**1. PRDの設計原則テーブルに「操作的定義」と「検証状態」を必須化する。**

```
| 原則 | 内容 | 根拠 | 操作的定義 | 検証状態 |
|---|---|---|---|---|
| Strain/Recovery | 独立2軸 | RESTQ着想 | Strain = stressのZ-score... | ドメインレビュー済み |
| 相関パターン | 30日閾値 | 検出力 | Pearson r≥0.5。自己相関でpower=50-62% | 検証済み: PG1では許容 |
```

**2. API ↔ US振る舞い照合**をSpecs導出後に追加する。

USが「〜を表示する」と書いている場合、APIがそのデータを返せるか確認する。
特に「データなし」「未入力」「未接続」のエッジケースで矛盾が起きやすい。

**3. 数値的主張には `[検証済み]` / `[未検証]` タグ**を付ける。

## Issue 3: レビューが任意なので、やらずに通過できてしまう

**深刻度:** Critical
**影響:** `.claude/rules/workflow.md`

### どういう問題か

テンプレートには8つのレビュー用Agent Teamsが定義されている。
しかし**ワークフローのどこでレビューを実行するか**が書かれていない。
レビューは完全に任意。やらなくても次に進める。

### 何が起きたか

設計を一気通貫で完了した時点では、問題はゼロだと思っていた。
ユーザーから「レビューして」と言われて初めて実施 → 12件発見。
その後さらに3ラウンドで21件追加。

**Issue 1〜2の問題はすべて、レビューゲートがあれば早期に発見できた。**

### こうすれば防げた

ワークフローの各フェーズにレビューゲートを組み込む:

```
MRD完了 → review-mrd → Pass後にPRDへ
PRD完了 → review-prd → Pass後にConceptual Modelへ
Conceptual Model完了 → review-model → Pass後にScreensへ
Specs + Test Spec完了 → review-specs → Pass後にImpl Planへ
実装完了 → review-code → Pass後にマージ

Pass条件: Critical 0件。Warning以下は次フェーズと並行で修正可。
```

レビューチーム構成の指針:
- **ドメイン専門家を含める**。PG1ではメンタルヘルスの観点がStrain構成の誤りを発見した
- review-specsは最低限「技術」「機能」「ドメイン」の3観点

## Issue 4: 並列OKと書かれたトラックに実は依存がある

**深刻度:** High
**影響:** `.claude/rules/workflow.md`

### どういう問題か

ワークフローは「Track A（Specs）とTrack B（US → Test Spec）を並列で進める」と書いている。
しかし**Test SpecはAPI Spec・Auth Specを参照する**。並列実行すると、Test Specが未完成のAPI Specを元に書かれてしまう。

### 何が起きたか

- Test Specが`GET daily_records`と書いたが、API Specは`GET /api/v1/trends`だった
- Test Specに「auth-spec具体化後に更新する」というプレースホルダが残った

### こうすれば防げた

```
Track A: Specs導出（順次）— DB Schema → API Spec → Auth Spec → UI Spec
Track B: User Stories（Track Aと並列可）
Track C: Test Spec（Track A完了後）
```

User Storiesの記述はSpecsと並列で進められるが、Test SpecはAPI/Auth Specへの参照があるため分離する。

## Issue 5: UI Specにライブラリ選定の枠がない

**深刻度:** High
**影響:** `docs/specs/ui-spec.md`（テンプレート構造）

### どういう問題か

UI Specはデザインシステム（Material Web）を参照するが、**デザインシステムにない要素**（グラフ、マップ、リッチエディタ等）のライブラリ選定セクションがない。

### 何が起きたか

- トレンドグラフにChart.jsが必要だったが、未選定のまま設計完了
- フィジカル（0-100）とメンタル（1-7）のY軸スケール破綻が見落とされた

### こうすれば防げた

UI Specテンプレートに「外部ライブラリ依存」セクションを追加:

```
| 用途 | ライブラリ | 選定理由 | バンドルサイズ |
|------|----------|---------|-------------|
| グラフ | Chart.js + annotation plugin | タップイベント、カラーゾーン帯 | ~65KB gzip |
```

product-model.jsonのscreensを走査し、デザインシステム外の要素がある場合にライブラリ選定を必須化。

## Issue 6: LLMが書いた文献引用が不正確でも気づけない

**深刻度:** High（2回再発）
**影響:** `.claude/rules/workflow.md`, `docs/reqs/prd.md`（テンプレート構造）

### どういう問題か

PRDの設計原則に`根拠`列があり、学術文献を引用する。
LLMはもっともらしい引用を生成するが、**正確性を検証するステップがワークフローにない**。
不正確な根拠がそのまま下流の設計判断に使われる。

### 何が起きたか

**文献引用の不正確（8箇所）:**

| LLMが書いた内容 | 実際 |
|---|---|
| 「VASより脱落率が低い」(Haslbeck 2025) | コンプライアンスに有意差なし |
| 「EMA疲労閾値15項目、5分」 | 出典なし。LLMの捏造 |
| Hedges' g = -0.776 を「集中」に帰属 | 「単純注意の失敗」が正 (Lim & Dinges 2010) |
| 「主観ファースト」は実証済み | 設計仮説。実証未確認 |
| 「ウェアラブルはZ-score正規化」 | 個人ベースラインは使うがZ-scoreとは限らない |

**統計的数値の過大評価:**
- 「Pearson power 50-60%」→ 精密計算で44.5%
- 「30日で十分」→ 自己相関考慮でpower=50-62%（不十分）

### こうすれば防げた

**1. PRDの`根拠`列に検証ステータスを追加する:**

```
| 原則 | 根拠 | 検証 |
|---|---|---|
| トレンド重視 | Baron et al. 2017 | ✅ 査読済み |
| 最小入力 | Wrzus & Neubauer 2023 | ✅ 査読済み |
| 相関30日閾値 | 検出力計算 | ⚠️ PG1では許容 |
```

**2. LLMが生成した引用は原典との照合を必須化する。**
Issue 3のレビューゲート（review-prd）に文献査読を含める。

## 対応の優先順位

```
                   Issue 3 (レビューゲート)
                   ← 最初に入れる。他の全Issueの早期発見に効く
                        │
          ┌──────────────┼──────────────┐
          │              │              │
     Issue 1         Issue 2        Issue 6
   (依存追跡)    (抽象→具体)    (文献検証)
   3回再発        3回再発        2回再発
          │              │
     Issue 4         Issue 5
  (並列トラック)  (UI技術選定)
```

**Issue 3を先に入れれば、他のIssueの問題は「レビューで捕まる」状態になる。**
その上でIssue 1→2→6の順にワークフローを強化する。Issue 4と5はテンプレート構造の修正のみ。
